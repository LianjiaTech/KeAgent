## 🚨 您必须遵守的关键规则

### Blender API 规则
- **强制**：尽可能优先选择数据 API 访问（`bpy.data`、`bpy.types`、直接属性编辑），而不是脆弱的上下文相关的 `bpy.ops` 调用；仅当 Blender 主要作为操作符公开功能时才使用“bpy.ops”，例如某些导出流
- 操作员必须失败并显示可操作的错误消息——永远不要在让场景处于模糊状态时默默地“成功”
- 干净地注册所有类并支持在开发过程中重新加载，没有孤立状态
- UI 面板属于正确的空间/区域/类别 - 切勿在随机菜单中隐藏关键的管道操作

### 无损工作流程标准
- 在没有明确用户确认或空运行模式的情况下，切勿破坏性地重命名、删除、应用转换或合并数据
- 验证工具必须在自动修复问题之前报告问题
- 批处理工具必须准确记录它们更改的内容
- 导出器必须保留源场景状态，除非用户明确选择进行破坏性清理

### 管道可靠性规则
- 命名约定必须是确定性的并记录在案
- 变换验证分别检查位置、旋转和缩放 - “全部应用”并不总是安全的
- 当下游工具依赖于槽索引时，必须验证材料槽顺序
- 基于集合的导出工具必须具有明确的包含和排除规则 - 无隐藏场景启发法

### 可维护性规则
- 每个附加组件都需要明确的属性组、操作员边界和注册结构
- 会话之间重要的工具设置必须通过“AddonPreferences”、场景属性或显式配置保留
- 长时间运行的批处理作业必须显示进度并且在可行的情况下可以取消
- 如果一个简单的清单和一个“修复所选”按钮就可以了，就避免使用聪明的用户界面
